# 12. 運用手順書

## 12.1 目的

本書は、アプリケーションの日常運用、障害発生時の対応、および定期メンテナンス手順を定義する。

------------------------------------------------------------------------

-   config.yaml が配置されている
-   template.html が存在する
-   config/credentials.json が配置されている
-   初回のGoogleログイン・スコープ同意が完了し config/token.json が生成されている
-   WordPress Application Password 設定済み

## 12.4 通常運用フロー

1.  Gmail未読メール取得（MIME添付を`MailData.attachments`として取得。各メールの
    受信時刻`internalDate`も取得する）
2.  受信時刻の古い順へ並び替え（Gmail一覧の返却順は信頼しない）。以降、この順で
    1件ずつ直列処理する
3.  話数解決（明示話数の使用、または自動採番。12.4.2参照）
4.  添付画像をAttachmentManagerがローカル保存（対応形式: JPG / JPEG / PNG、大文字小文字区別なし）
5.  画像並び替え（IMG9999→IMG0001のロールオーバー考慮）
6.  WordPressへ画像アップロード（並び替え順を維持、先頭画像をアイキャッチに設定）
7.  前回記事解決（同一作品名の直前話のみ。12.4.3参照）
8.  記事作成（下書き／予約投稿）
9.  投稿成功記事を実行中キャッシュへ登録（この実行中だけ有効。次回起動へは引き継がない）
10. 投稿履歴保存（投稿成功時のみ）
11. メールへラベル付与・既読化（将来のgmail.modifyスコープ対応時）
12. 次のメールへ（失敗しても受信時刻順のまま処理を継続する）

対応形式以外の添付（例: .txt, .pdf等）や、対応画像が1枚も無いメールは
投稿・履歴保存を行わずスキップする。WordPress投稿に失敗した場合も履歴・実行中キャッシュに
登録しない（次回実行時に再試行される。後続メールは受信時刻順のまま処理を継続する）。

## 12.4.1 投稿メール件名とWordPress投稿タイトルの形式

投稿メールの推奨件名は次の形式とする。

    作品名 ~ 話数

例

    お菓子外しさんとカタカナ男 ~ 42

以下の表記揺れも話数として認識し、標準形式へ正規化する。

    作品名 42
    作品名　４２（全角スペース・全角数字）
    作品名 ~ 42
    作品名　〜　４２（全角スペース・波ダッシュ・全角数字）
    作品名 No42（旧No形式。No/no・Ｎｏ／ｎｏ、No後のスペース有無を問わない）
    作品名 No 42
    作品名 Ｎｏ４２

件名末尾の話数表記は、次の優先順で判定する。

1.  `~`・`〜`・`～`区切りの明示話数
2.  半角/全角`No`形式の明示話数
3.  区切りなし（空白のみ）の裸の数字

この順で判定することで、「作品名 No 42」のような`No`区切りが「区切りなしの裸の数字」
として誤って解釈され、`No`まで作品名に取り込まれてしまうこと（誤: 作品名
「作品名 No」・話数42）を防いでいる。

件名の記載形式によらず、WordPress上の投稿タイトルは必ず次の標準形式へ変換される。

    作品名 ~ 話数

作品名の途中に含まれる数字は話数として扱わない。話数として扱うのは件名末尾の
独立した数字のみ。

**話数省略時**：件名に話数が無い場合は、件名全体を作品名として扱い、WordPress上の
同一作品名の最新記事を検索して最大話数+1を自動採番する。既存記事が1件も無い場合は
第1話とする（採番時は標準形式`作品名 ~ 話数`および互換のため旧`No`形式の記事を対象とし、
別作品名や過去に誤った形式で作成された記事は採番対象に含めない）。

話数の解決（明示話数の使用、または自動採番のための最新記事検索）は、画像アップロード・
カテゴリ作成より前に行う。最新話数検索に失敗した場合は、WordPressへ画像やカテゴリを
作成しない。

空の件名、または「作品名 ~」「作品名 No」のように区切り表記だけが残り話数が存在しない
件名は無効としてスキップし、理由をWARNログへ記録する。

------------------------------------------------------------------------

## 12.4.2 複数メール受信時の話数自動採番

話数省略メールが複数同時に届いた場合も、12.4節の受信時刻順に1件ずつ処理しながら
自動採番する。話数は次の2つの最大値を比較し、大きい方+1とする。

1.  WordPress上に既に存在する同一作品名の最大話数
2.  同一実行中に投稿成功した同一作品名の最大話数（実行中キャッシュ）

WordPress側の検索結果が直前に作成した記事をすぐ反映しない場合でも、実行中キャッシュを
参照することで、同一実行中に話数が重複して採番されることはない。既存記事・実行中記事の
いずれも無ければ第1話とする。

## 12.4.3 前回記事リンクの解決

現在の話数が`N`の場合、前回記事は同一作品名の`N-1`話だけを対象とし、次の優先順で解決する。

1.  同一実行中の投稿成功キャッシュ（同一作品・`N-1`話）
2.  WordPress上の標準形式「作品名 ~ N-1」
3.  WordPress上の旧形式「作品名 NoN-1」「作品名 No N-1」（フォールバック）
4.  いずれにも一致しなければ前回記事なし

別作品の記事、同一カテゴリの最新記事、部分一致した記事を前回リンクとして採用することは
ない。第1話（`N=1`）では前回記事検索を行わない。直前話（`N-1`）が存在しない場合
（欠番）は、それより前の話数（`N-2`以前）へ遡ることはせず、前回セクションを本文から
完全に削除する。複数作品が同一実行に混在しても、前回リンクが作品間で交差することはない。

## 12.5 障害対応

  障害                   確認事項                        対応
  ---------------------- ------------------------------- ----------
  Gmail接続不可          credentials.json・token.json    設定修正・再認証
  WordPress認証失敗      Application Password            再発行
  画像アップロード失敗   API応答                         再実行
  テンプレートなし       ファイル有無                     配置

------------------------------------------------------------------------

## 12.6 ログ確認

ログファイル:

    logs/application.log

確認ポイント

-   ERROR
-   WARN
-   Retry回数
-   投稿ID
-   処理順（複数メール受信時。受信時刻・件名・Message-IDが記録される）
-   前回記事の解決結果（実行中キャッシュ／WordPress標準形式／旧形式／前回記事なし）
